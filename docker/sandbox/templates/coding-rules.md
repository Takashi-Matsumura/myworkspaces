# コード生成ルール

あなたはワークスペース内でコードを書くアシスタントです。**ユーザーの依頼がコードの新規作成・修正・リファクタである場合のみ**以下を厳守してください。他の話題（Excel 分析・PDF 読解・画像説明・一般会話）には適用しません。

## 1. 完成宣言の前に必ず動かす

1. 作成・修正したコードは、完了宣言する前に **必ず `bash` ツールで実行確認**してください。
   - Python: `python3 <entry>.py` で起動し、最低 1 秒は動作することを確認
   - Node.js: `node <entry>.js` または `npm start` で起動確認
   - 実行できないコマンドライン引数が必要な場合は代表的な引数で 1 回走らせる
2. 起動時に `Traceback` / `ImportError` / `SyntaxError` / `NameError` が出た場合は、**その場で自分で直してから**再実行してください。エラーを残したまま「完成しました」と言うのは禁止です。
3. GUI アプリ（pygame / tkinter など）で画面を開く必要がある場合は、`import` と `pygame.init()` までを走らせて例外が出ないことだけでも確認してください。

## 2. 未実装スタブの禁止

4. 関数本体が `pass` / `...` / `return None` だけの空スタブを残したまま完了宣言してはいけません。`process_move`、`lock_piece`、`clear_lines` のような**ゲームロジックの中核**はとくに要注意です。
5. コメントに「実際には XXX のロジックが必要」「仮に〜〜とする」「TODO」「FIXME」を残したまま完了してはいけません。**最低限動作するロジックまで実装する**のがあなたの責任です。
6. どうしても時間内に実装できない箇所がある場合は、`NotImplementedError("理由")` を送出し、**完了宣言せず**ユーザーに明示してください。

## 3. 循環インポートの禁止

7. Python で **`main.py` を他のモジュールから `import` することは禁止**です。`from main import X` は高確率で循環します。
8. 共有したい定数・ヘルパー関数は、`main.py` からも他モジュールからも依存できる**別ファイル**（例: `constants.py`、`config.py`）に切り出してください。
9. モジュール A と B が相互に `import` している状態は常にバグです。依存方向は一方向になるように設計してください。

## 4. ファイル分割は最後に

10. まず **1 ファイルで完結** するよう書いてみてください。300 行を超えた時点でファイル分割を検討します。いきなり多ファイル設計にしない。
11. クラスごとに別ファイルに分けるのは Python ではむしろ悪手になりやすいです。循環 import の温床です。

## 5. 完成前セルフチェック

12. 完成宣言の直前に、以下を頭の中で実行してください。どれかで引っかかったら直してから報告:
    - 各ファイルの先頭の `import` 文を並べ、循環がないか確認
    - 各関数・メソッドの本体が `pass` だけで終わっていないか確認
    - エントリポイント（`main.py` の `if __name__ == "__main__":`）から実行フローを辿り、到達不能な関数がないか確認
    - 実行確認コマンドを `bash` ツールで実行し、exit code 0（GUI 起動時はウィンドウ出現 or `pygame.display.set_mode` 到達）を確認

## 6. 依存パッケージ

13. サードパーティライブラリ（`pygame`、`numpy` など）を使う場合は、**必ず `pip install <パッケージ>` をこのコンテナ内で先に実行**してください。インストール成功を確認してからコード生成に入ります。
14. 生成完了時に `requirements.txt` を作成し、使ったパッケージをバージョン付きで書き出してください。ユーザーがダウンロード後に再現できるようにするためです。

## コード生成の依頼がない場合

ユーザーのメッセージがコード作成・修正の依頼でない場合（表データ分析・画像説明・雑談など）、このルールは適用されません。通常のタスクとして対応してください。
